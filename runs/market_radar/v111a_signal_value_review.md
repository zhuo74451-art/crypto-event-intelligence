| 字段 | 价值评分 | 理由 |
|------|---------|------|
| **source_type** | ⭐⭐⭐ | 数据来源标识（api/real/fixture），对系统治理有价值，对终端用户意义不大。但作为 trust gate 的输入是必需的。 |

### 低价值 / 噪音字段

| 字段 | 价值评分 | 理由 |
|------|---------|------|
| **funding** | ⭐⭐ | 理论上有价值（期货市场情绪），但当前所有发送卡 funding 均为 0.00%（年化 0.1%-1.4%），从未出现过异常值。在卡片中展示为"+"或"0.00%"纯属占位，没有信息增量。需要设置阈值（如 funding > 0.01% 或 < -0.01%）才值得展示。 |
| **open_interest** | ⭐⭐ | 展示了 OI 绝对值，但没有给出"相对变化"或"历史百分位"。用户在卡片中看到 "OI: 18.26 亿美元" 无法判断这是多还是少。需要加入 OI 变化率（如 24h OI delta%）或历史分位数才有意义。 |
| **volume** | ⭐⭐ | 同样只展示绝对值，无相对比较。对 BTC 来说 63 亿美元日成交量是正常还是放大？卡片没有回答这个问题。 |
| **fallback_used** | ⭐ | 系统级字段，对终端用户零价值。不应出现在用户可见输出中。 |

### 建议保留字段（用户可见卡片）

1. **asset** — 必须保留
2. **price_change_pct** — 保留，但需增加波动分位数对标
3. **signal_type** — 保留（用于信号分类）
4. **source_type** — 保留（用于 trust 链路，但不显示给用户）
5. **OI delta%** (新增) — 替代当前 OI 绝对值
6. **volume surge ratio** (新增) — 替代当前 volume 绝对值
7. **funding alert** (有条件展示) — 仅当 funding 绝对值 > 阈值时展示

### 建议弱化字段

1. **funding** — 仅当异常时展示，否则隐藏
2. **open_interest (绝对值)** — 改为 OI delta%
3. **volume (绝对值)** — 改为 volume surge ratio

### 建议删除或后置字段

1. **fallback_used** — 不应出现在用户可见卡片中（系统日志保留即可）
2. **"是否拥挤"** — 当前始终显示"否"，无实际判断逻辑，属于占位文字

当前 `market_anomaly` 的触发条件为：

- 信号来源：Hyperliquid API
- 信号类型：market_anomaly（基于 24h 价格变化）
- 准入门：所有符合条件的 market_anomaly 信号均可发送（最多 3-5 张/批次）

### 7.2 建议准入优化

| 建议 | 优先级 | 说明 |
|------|--------|------|
| 增加同资产冷却期 | P0 | 同一 asset 在 N 分钟（建议 30-60min）内只能发 1 次。当前 ARB 被重复发了 3 次（2245→2250 仅隔 11 分钟）。 |
| 增加波动分位数过滤 | P0 | 仅当 24h 跌幅超过该资产历史的 90th/95th 分位数时才触发。避免对小币种的日常波动过度播报。 |

| BTC 加权 | P2 | BTC 异动天然比山寨币更有市场级意义，可考虑降低 BTC 的触发阈值（如 -4% 即可触发）或增加 special badge。 |

## 8. 是否建议继续技术扩张

**不建议继续技术扩张。当前阶段应聚焦内容质量优化。**

理由：
1. 主链路（Hyperliquid API → market_anomaly → render → gate → TGTransport）已验证可靠，技术债务可控。
2. RSS / onchain_position / whale_transfer / risk_alert 四个数据源处于暂缓状态（v1.10-J 已明确），修复成本未知且价值未经验证。
3. 当前最紧迫的问题不是"能发多少种卡片"，而是"发出的每张卡片是否有足够的情报密度"。
4. 在内容质量未达标之前扩张数据源，会导致更多低质量卡片涌入测试频道，稀释用户的关注度。

**建议的优先顺序：**
1. 内容质量优化（本复盘 → v1.11-B 准入规则实现）
2. 测试频道用户反馈收集（如有真实用户）
3. 再考虑是否恢复 RSS / onchain_position / whale_transfer 数据源

## 9. 是否建议保持正式频道冻结

**强烈建议保持正式频道冻结，至少在以下条件满足前不解冻：**

1. ✅ 同资产冷却期机制已实现
2. ✅ 波动分位数过滤已上线
3. ✅ funding 异常联合触发已实现（有条件展示）
4. ✅ OI delta% 替代 OI 绝对值
5. ✅ 连续 3 批次（≥9 张卡片）中噪音率 < 30%（即 ≥70% 的卡片被判断为"有情报价值"）
6. ✅ 正式频道首张卡片经人工确认后再继续

当前状态下解冻正式频道的风险：
- 用户会被"ARB 跌了 5.96% → 15 分钟后又跌了 6.75%"这类重复信息淹没
- 频道信号噪音比会很低，损害用户对频道的信任
- 如果早期用户体验差，后续即使内容质量提升也难以挽回

## 10. v1.11-B 推荐方向

### Option A：提高 market_anomaly 准入门槛
**内容**：将跌幅阈值从当前值（约 5%）提高到 8-10%，或加入波动分位数过滤。

**优点**：
- 实现成本最低（改一个阈值参数）
- 立即减少低质量卡片数量
- 不需要架构变动

**缺点**：
- 可能漏掉"跌幅不大但伴随 OI/Volume 异常"的复合信号
- 单维度优化，未解决"辅助字段无信息增量"的核心问题
- 对于 BTC 来说 8-10% 的日跌幅非常罕见，可能导致 BTC 卡片几乎不发

**推荐度**：⭐⭐⭐⭐（第 2 推荐）

---

### Option B：增加"多因子异常"规则
**内容**：不只看跌幅，同时要求 OI delta% 超阈值 或 Volume surge 超阈值 或 funding 异常，形成 2/4 或 3/4 的多因子联合触发。

**优点**：
- 从根本上解决"单调跌幅播报"问题
- 每张发出的卡片至少 2 个因子异常，情报密度显著提升
- 与当前架构兼容（Card Router、SignalTrustGate、pre_send_gate 不需要重构）

**缺点**：
- 实现成本中等（需新的数据采集：OI delta%、Volume surge ratio、funding alert）
- 可能显著减少发送量（初期可能 0 发送，需要调参）
- 需要回测历史数据确定合理阈值

**推荐度**：⭐⭐⭐⭐⭐（第 1 推荐）

---

### Option C：保持当前规则，但降低发送频率
**内容**：规则不变，但限制发送频率（如每 30 分钟最多 1 批次，每天最多 N 批次）。

**优点**：
- 实现成本最低
- 减少用户收到的卡片总数

**缺点**：
- 治标不治本：卡片内容质量问题完全不解决
- 用户收到更少的"无信息增量卡片"而已，每张卡片品质不变
- 如果 30 分钟内只有噪音信号，用户还是会收到噪音

**推荐度**：⭐⭐（第 3 推荐）

---

### Option D：只在测试频道继续观察，不做正式频道准备
**内容**：维持现状，继续测试频道发送，不做任何改动，不做正式频道准备。

**优点**：
- 零成本
- 积累更多真实发送数据

**缺点**：
- 没有进展，无法提升内容价值
- 测试频道继续接收低质量卡片
- 永远停留在"行情播报器"阶段

**推荐度**：⭐（第 4 推荐，仅作为过渡方案）

---

### 推荐排序

| 排序 | 选项 | 理由 |
|------|------|------|
