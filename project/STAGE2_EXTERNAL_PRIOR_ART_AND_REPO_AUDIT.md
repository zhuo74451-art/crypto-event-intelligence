# Stage 2 — External Prior Art Challenge and Repository Audit

**Status:** Active

## Purpose

Find the smallest reliable implementation route for the accepted autonomous-cognition responsibility. Do not assume the current code or the proposed agent decomposition is the best foundation.

## Workstream A — External prior art challenge

Evaluate mature methods and components for:

- event sourcing and immutable revisions;
- durable workflows, scheduling, retries and recovery;
- state machines and legal-transition enforcement;
- evidence provenance and content addressing;
- entity resolution and knowledge graphs;
- retrieval, research and source-health systems;
- structured agent outputs and model gateways;
- selective prediction, abstention and calibration;
- time-ordered evaluation and leakage prevention;
- observability, resource accounting and kill controls.

For each candidate record:

- real capability and maintenance status;
- license and integration boundary;
- what responsibility it can own;
- what still requires local invention;
- operational complexity and failure modes;
- adopt, adapt, reference or reject decision.

## Workstream B — Design challenge

Test simpler alternatives against the accepted outcome:

1. deterministic pipeline with one semantic model;
2. deterministic pipeline with Thesis and Risk passes only;
3. full bounded-role architecture;
4. event-only watchlist without persistent theses;
5. thesis lifecycle without directional judgment;
6. human-free operation with no multi-agent conversation.

The preferred design is the least complex route that preserves evidence integrity, useful abstention, lifecycle maintenance, recovery and owner outcomes.

## Workstream C — Existing repository audit

Inspect `main`, Draft PR #16, tests, artifacts, runtime entrypoints and documentation.

Classify every relevant component:

- `RETAIN` — already fits the accepted responsibility;
- `ADAPT` — useful but requires bounded change;
- `QUARANTINE` — may hold evidence but must leave the active path;
- `REMOVE` — conflicts with the product or adds unjustified complexity;
- `MISSING` — required responsibility has no implementation.

Audit surfaces:

- acquisition and evidence contracts;
- event identity and history;
- market state and expectation context;
- research claims;
- strategy components;
- arbitration and packets;
- storage and revision history;
- unattended lifecycle and scheduling;
- resource governance;
- status, stop, recovery and observability;
- replay, shadow and evaluation artifacts.

## Required output

- external-foundation comparison;
- simplest viable architecture decision;
- active-path map;
- responsibility coverage matrix;
- retain/adapt/quarantine/remove/missing register;
- obsolete code and test candidates;
- missing contracts and evidence;
- engineering dependency graph;
- first complete delivery slice;
- explicit non-goals and stop conditions.

## Acceptance

Stage 2 exits only when:

- mature foundations have been checked rather than guessed;
- at least one simpler architecture has challenged the preferred design;
- every active-path component has a classification and evidence reference;
- the smallest complete path is identified;
- current code claims match remote reality;
- implementation can begin without unresolved product or responsibility decisions.
